# 目录结构说明

## 总体原则

这个仓库按“设计 -> 实现 -> 验证 -> 交接”的闭环组织，而不是按随手堆文件的方式组织。

每个目录都应该承担单一职责，避免“看起来都能放”的灰色区域。

## 根目录

### `AGENTS.md`

仓库级工作约定。Codex 进入项目时优先读它，明确：

- 这个仓库是干什么的
- 应该先读哪些文档
- 客户端、服务端、共享代码怎么拆
- 推荐使用哪些技能和流程

### `README.md`

给人看的快速入口。用来帮助新成员理解：

- 这个仓库适合做什么
- 怎么开始用
- 文档去哪看

## `src/`

Roblox 代码主目录。

### `src/client/`

适合放：

- 输入处理
- 镜头和手感
- UI 控制
- 本地动画或表现逻辑
- 客户端预测或视觉反馈

不适合放：

- 最终奖励结算
- 存档写入
- 货币结算
- 信任客户端即可完成的关键玩法逻辑

### `src/server/`

适合放：

- 服务端权威逻辑
- 回合结算
- 奖励发放
- 存档和数据校验
- 反作弊检查
- Matchmaking 或房间管理

### `src/shared/`

适合放：

- 常量
- 枚举
- 数据结构定义
- Remote 协议说明
- 纯工具函数
- 共享配置表

## `design/`

设计文档目录。

### `design/gdd/`

项目级设计文档，面向整体：

- `game-concept.md`
- `systems-index.md`

### `design/systems/`

单个系统的详细设计文档，面向实现：

- 玩家控制
- 战斗
- 背包
- 经济
- 回合流程
- 匹配

一个原则：只有“当前真的要实现或已经在频繁修改”的系统，才值得写成单独文档。

## `docs/`

辅助说明目录。

适合放：

- 使用说明
- 工作流说明
- 项目定位说明
- 模板

### `docs/templates/`

所有模板都放这里，避免实际文档和模板混在一起。

## `production/`

面向持续协作和会话接力的目录。

### `production/session-state/active.md`

这是最重要的协作文件之一。它不是长篇总结，而是短而有用的“下次接着干”记录。

建议只记录：

- 当前任务
- 关键文件
- 已做决定
- 风险和假设
- 下一步

## `tests/`

测试和验证文档目录。

### `tests/playtest/`

适合放：

- Playtest 报告
- Bug 复现记录
- 改动后的回归检查结果

## `scripts/`

本地辅助脚本目录。

当前已经包含：

- `install-codex-skills.ps1`
- `check-project.ps1`

后续可以继续补：

- 初始化 Rojo 结构脚本
- 文档检查脚本
- 打包或导出脚本

## `codex/`

Codex 专用内容目录。

### `codex/skills-src/`

这里放仓库自己维护的 skills 源文件，而不是把 skills 散落在全局目录里。这样做的好处是：

- 技能可以跟仓库版本一起管理
- 团队成员拿到仓库就能安装同一套 skills
- 修改 skill 时有明确的源码位置
