# 工作流总览

## 目标

这套工作流的目标不是把所有事情都流程化，而是保证 Roblox 项目最容易失控的几个环节可以形成闭环：

- 方向是否清楚
- 系统是否拆清楚
- 代码是否有来源依据
- 改完之后是否被验证
- 下次是否能继续接着做

## 标准流程

### 第 1 步：盘点当前状态

使用 `roblox-start`。

适用场景：

- 新项目刚起步
- 很久没打开仓库
- 不确定当前最合理的下一步是什么

输出：

- 当前项目状态判断
- 一个最推荐的后续动作

### 第 2 步：完成概念文档

完善 `design/gdd/game-concept.md`。

这一步的目标不是把一切写满，而是先回答 4 个问题：

- 这个游戏是什么
- 玩家反复玩什么
- 为什么值得做
- MVP 最小做到什么

### 第 3 步：生成系统总表

使用 `roblox-map-systems`。

输出 `design/gdd/systems-index.md`，明确：

- 系统有哪些
- 哪些是 MVP
- 哪些归客户端、服务端、共享层
- 哪些依赖关系会决定实现顺序

### 第 4 步：写单系统设计文档

使用 `roblox-design-system`。

不是所有系统都要立刻写，优先写：

- 下一个准备实现的系统
- 风险最高的系统
- 会影响多个模块边界的系统

### 第 5 步：按文档落地代码

使用 `roblox-implement-feature`。

这一步的原则是：

- 只改和当前功能直接相关的代码
- 明确 client / server / shared 的边界
- 改完要说明哪些地方已验证，哪些地方还没法验证

### 第 6 步：做验证和回归

使用 `roblox-playtest-report`。

输出到 `tests/playtest/`，关注：

- 是否能复现问题
- 实际表现与预期是否一致
- 控制台有没有异常
- 是否还存在高优先级风险

### 第 7 步：更新会话状态

更新 `production/session-state/active.md`。

这一步的作用是保证：

- 下次回来不会丢上下文
- 其他协作者也能快速接手

## 两种常见起步方式

### 方式 A：从 0 到 1

适合完全新项目：

1. `roblox-start`
2. 填写 `game-concept.md`
3. 运行 `roblox-map-systems`
4. 运行 `roblox-design-system`
5. 运行 `roblox-implement-feature`
6. 运行 `roblox-playtest-report`

### 方式 B：已有代码，补结构

适合已经开工但文档混乱的项目：

1. `roblox-start`
2. 反写或补写 `game-concept.md`
3. 生成 `systems-index.md`
4. 只为正在频繁改动的系统补写设计文档
5. 再继续实现和验证

## 一个实用原则

不要试图一次把所有文档写完整。

最有效的方式通常是：

- 方向层面：先写概念文档
- 结构层面：再写系统总表
- 实现层面：只写当前要做的系统文档

这样能让结构真正服务开发，而不是反过来拖慢开发。
