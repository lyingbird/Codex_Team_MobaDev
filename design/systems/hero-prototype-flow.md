# 英雄原型流程

> **状态**：草稿
> **归属区域**：Server + Shared
> **最后更新**：2026-03-24

---

## 系统概述

英雄原型流程系统不是传统意义上的正式对局流程，而是一个“为了验证英雄体验而存在的战斗原型循环”。它负责让玩家能够快速进入测试、释放技能、观察反馈、死亡后快速重来，并尽可能缩短每次验证手感的等待时间。

## 玩家体验目标

玩家进入这个原型模式后，应该在很短时间内就能开始战斗，不需要等待兵线、塔、防御目标或完整地图节奏。它的目的就是让玩家持续重复：

- 移动
- 放技能
- 命中
- 躲技能
- 死亡
- 重开

这样才能高频验证英雄是否好玩。

## 核心职责

- 管理原型战斗场景的进入与重置
- 负责快速开始、快速重生、快速重新测试
- 承接训练假人、简易敌对目标或 1v1 原型对抗
- 提供基础测试信息，如当前英雄、技能冷却、最近伤害结果

## 输入

- 玩家进入测试场
- 玩家选择英雄
- 死亡事件
- 重置请求
- 测试模式配置，如训练假人、单人测试、双人对拼

## 输出

- 角色出生
- 场景重置
- 快速重生
- 测试状态同步
- 最近一次技能结果或伤害结果展示

## 规则与状态变化

1. 玩家进入原型模式后，应在极短时间内生成英雄并获得完整技能权限。
2. 原型流程默认不依赖正式比赛人数条件。
3. 当玩家死亡后，进入短暂等待，再快速重生到固定测试位置。
4. 玩家可以主动触发重置，用于重新测试同一组技能。
5. 训练假人或简易敌对目标应在重置后恢复满状态。
6. 如果需要双人对拼，系统应支持快速回到“就位可再次开打”的状态。

## 服务端权威与复制策略

- 最终裁决方：Server
- 远程通信需求：
  - `PrototypeResetRequest`
  - `PrototypeStateSync`
  - `PrototypeRespawnNotice`
- 反作弊校验点：
  - 客户端不能伪造重置结果
  - 客户端不能跳过死亡或冷却限制
- 需要复制给客户端的状态：
  - 当前原型模式状态
  - 最近重置时间
  - 当前测试类型

## 数据模型

| 字段 | 类型 | 持有方 | 说明 |
| --- | --- | --- | --- |
| PrototypeMode | string | Server + Shared | Training、Dummy、Duel 等 |
| SpawnPointId | string | Server + Shared | 当前出生点 |
| LastResetTime | number | Server | 最近重置时间 |
| RespawnDelay | number | Shared | 重生等待时间 |
| SelectedHeroId | string | Server + Shared | 当前测试英雄 |

## 玩家交互流程

1. 玩家进入测试场并选择英雄
2. 系统生成英雄，玩家立即可以移动与施法
3. 玩家与假人或目标交互，观察技能手感和反馈
4. 死亡或主动重置后，系统快速清场并重新开始

## 边界情况

- 玩家在技能前摇时点击重置
- 玩家死亡瞬间目标也被击杀
- 双人对拼中一方掉线
- 训练假人状态异常未重置

## 可调参数

| 参数 | 默认值 | 安全范围 | 用途 |
| --- | --- | --- | --- |
| 重生延迟 | 2 秒 | 0.5 到 4 秒 | 提高反复测试效率 |
| 场景重置延迟 | 1 秒 | 0 到 3 秒 | 清理投射物和状态 |
| 默认模式 | Training | Training / Dummy / Duel | 方便切换测试场景 |

## 监控与审核注意点

- 建议监控指标：平均测试时长、重置频率、技能释放频率、死亡后重开率
- 容易被滥用的点：频繁重置刷状态、利用重置跳过冷却

## 验收标准

- [ ] 玩家进入原型场后能在短时间内开始测试英雄
- [ ] 死亡和主动重置都能快速回到可继续测试的状态
- [ ] 原型流程不依赖兵线、塔、防御目标或正式比赛人数

## 未决问题
