# 技能释放与冷却

> **状态**：草稿
> **归属区域**：Client + Server + Shared
> **最后更新**：2026-03-24

---

## 系统概述

技能释放系统负责承接玩家从“按下技能键”到“技能真正生效”之间的完整流程，包括预览、确认、前摇、释放、生效、后摇、冷却、打断和失败反馈。它是 MOBA 手感最关键的核心系统之一，比纯数值伤害更早决定玩家会不会觉得“这个英雄顺手”。

## 玩家体验目标

玩家应该明确感受到：

- 我按下技能键时，系统立刻给了我反馈
- 我知道这个技能现在是在预瞄、前摇还是已经放出去
- 如果技能失败，我知道为什么失败
- 如果技能可以打断、取消或缓冲，我能稳定感知这个窗口
- 在弱网环境下，技能也尽量“先有本地响应，再由服务端最终裁决”

换句话说，技能释放系统的目标不是单纯“能放技能”，而是让技能具备 MOBA 应有的节奏感、确认感和可预期性。

## 核心职责

- 接收技能输入
- 根据技能类型进入正确的施法阶段
- 处理技能预览、目标确认和本地反馈
- 向服务端提交施法请求并等待裁决
- 在服务端处理前摇、释放时刻、后摇、冷却与打断
- 向客户端广播施法结果、冷却状态和失败原因

## 技能类型范围

MVP 阶段至少支持 5 类技能：

1. 即时自施法
2. 指向型直线技能
3. 地面范围技能
4. 锁定目标技能
5. 位移技能

后续再扩展：

- 引导型技能
- 蓄力型技能
- 多段技能
- 可再次触发技能
- 召唤物技能

## 施法阶段

技能统一遵循下面的状态流，不同技能只是在部分阶段可跳过或停留时间不同：

1. `Ready`
   - 技能可用，等待输入
2. `Preview`
   - 玩家正在预瞄或选目标
3. `Buffered`
   - 玩家已提前输入，等待前一个动作结束后自动尝试施放
4. `Validating`
   - 客户端已提交请求，服务端正在校验
5. `Windup`
   - 技能进入前摇，技能尚未真正生效
6. `Firing`
   - 技能在这一时刻真正生成效果、伤害、位移或控制
7. `Recovery`
   - 后摇阶段，玩家可能不能立刻执行下一动作
8. `Cooldown`
   - 技能进入冷却
9. `Interrupted`
   - 技能在前摇或引导过程中被打断
10. `Failed`
   - 技能请求未通过，例如超范围、死亡、沉默、目标非法

## 手感优先原则

### 1. 输入要先响应，结果再校验

玩家一按下技能键，客户端必须立刻做出反馈：

- 按钮按下状态
- 指示器显示
- 目标线或范围圈
- 预估释放方向

不要等服务端回包后才出现第一个视觉反馈，否则技能会显得“黏”“慢”“按了没反应”。

### 2. 前摇必须可感知，但不能拖沓

MOBA 技能不是越快越好。好的技能手感来自“我能感觉到我要出手了”，而不是完全没有前摇。  
但前摇必须服务于角色风格，而不是单纯拖时间。

MVP 建议：

- 即时类技能：前摇 `0.05` 到 `0.12` 秒
- 主要伤害技能：前摇 `0.12` 到 `0.25` 秒
- 重技能或大招原型：前摇 `0.25` 到 `0.45` 秒

### 3. 命中确认必须清楚

技能真正生效时，要有足够明确的反馈：

- 动画关键帧
- 音效触发
- 投射物生成
- 地面提示落地
- 血条变化或受击反馈

玩家应该能区分：

- 技能已经按下
- 技能已经进入前摇
- 技能已经真正放出
- 技能已经命中或落空

### 4. 失败原因必须明确

当技能没有成功放出，必须能区分失败原因：

- 冷却中
- 法力或资源不足
- 目标超出范围
- 目标非法
- 英雄被控制
- 英雄已死亡
- 对局状态不允许

如果玩家只看到“按钮按了没反应”，这就是手感系统的失败。

### 5. 技能缓冲要保留，但不能失控

MOBA 技能系统通常要允许轻度输入缓冲，这样玩家在普攻后摇、前一个技能后摇里提前按下下一个技能时，不会因为窗口太苛刻而显得生硬。

MVP 建议：

- 技能缓冲窗口：`0.10` 到 `0.18` 秒
- 只保留单技能缓冲，不做复杂队列

### 6. 打断规则必须统一

玩家最怕“不知道为什么这个技能有时能放、有时被吞”。  
因此 MVP 阶段要先统一规则：

- 硬控可打断前摇
- 沉默可阻止新的技能释放，但不一定打断已放出的飞行物
- 位移中的技能是否可打断，要按技能配置显式声明
- 一旦技能进入真正 `Firing` 阶段，后续效果通常不应被撤销

## 输入与目标确认逻辑

### PC 端

- `Q/W/E` 对应技能键
- 鼠标位置决定指向或落点
- 锁定技能优先使用当前目标，否则尝试最近合法目标

### Mobile 端

- 点击技能键进入预瞄
- 拖动决定方向或落点
- 松手确认施放
- 支持最近目标辅助和吸附

### 统一规则

- 任何技能都必须定义最大施法距离
- 地面技能必须对目标点做最大范围裁切
- 指向技能必须对方向做标准化
- 锁定技能必须校验目标仍然合法

## 服务端权威与复制策略

- 最终裁决方：Server
- 客户端职责：
  - 输入采集
  - 预瞄和指示器
  - 本地立即反馈
  - 展示服务端同步回来的正式状态
- 服务端职责：
  - 校验施法条件
  - 决定技能是否进入前摇
  - 决定技能何时真正生效
  - 写入冷却、打断和释放结果

建议使用的远程事件：

- `AbilityCastRequest`
- `AbilityCastStarted`
- `AbilityCastResolved`
- `AbilityCastFailed`
- `AbilityStateSync`

## 规则与状态变化

1. 玩家按下技能键后，客户端先进入 `Preview` 或直接进入 `Validating`。
2. 如果技能需要选方向或落点，客户端先显示预览，再在确认时发请求。
3. 服务端校验通过后，技能进入 `Windup`。
4. 到达 `Firing` 时刻后，技能真正生成效果。
5. 技能进入 `Recovery`，然后写入 `Cooldown`。
6. 若在允许打断的阶段遭遇硬控或非法状态，则进入 `Interrupted`。
7. 若请求一开始就不合法，则进入 `Failed`，并返回失败原因。

## 数据模型

| 字段 | 类型 | 持有方 | 说明 |
| --- | --- | --- | --- |
| AbilityId | string | Shared | 技能唯一标识 |
| Slot | string | Shared | Q / W / E |
| CastType | string | Shared | 技能类型 |
| Stage | string | Shared | 当前施法阶段 |
| CooldownEndTime | number | Server + Shared | 技能冷却结束时间 |
| WindupSeconds | number | Shared | 前摇时长 |
| RecoverySeconds | number | Shared | 后摇时长 |
| BufferWindowSeconds | number | Shared | 输入缓冲窗口 |
| CurrentTargetId | string? | Client + Shared | 当前目标 |
| TargetPosition | Vector3? | Client + Shared | 当前瞄准点 |
| FailureReason | string? | Server + Shared | 最近失败原因 |

## 玩家交互流程

1. 玩家按下技能键
2. 客户端立即显示按钮反馈与指示器
3. 玩家确认方向、目标或落点
4. 客户端向服务端发送施法请求
5. 服务端校验成功后广播技能开始
6. 技能进入前摇并在指定时刻正式生效
7. 客户端显示释放、命中、冷却和失败反馈

## 边界情况

- 技能前摇时被眩晕
- 技能确认时目标死亡
- 地面技能目标点超出最大范围
- 玩家在移动端拖拽时误触取消
- 玩家弱网导致本地已经看到前摇，但服务端判定失败
- 技能在前一个动作后摇中被缓冲输入
- 连按技能导致重复请求

## 可调参数

| 参数 | 默认值 | 安全范围 | 用途 |
| --- | --- | --- | --- |
| 输入缓冲窗口 | 0.15 秒 | 0.10 到 0.18 秒 | 提高手感稳定性 |
| 目标吸附半径 | 10 | 6 到 16 | 帮助移动端和手柄用户 |
| 即时技能前摇 | 0.08 秒 | 0.05 到 0.12 秒 | 保证有反馈但不拖 |
| 普通技能前摇 | 0.18 秒 | 0.12 到 0.25 秒 | 核心手感区间 |
| 后摇时长 | 0.12 秒 | 0.05 到 0.25 秒 | 决定连招顺滑度 |
| 失败提示显示时长 | 0.6 秒 | 0.3 到 1 秒 | 帮助玩家理解失败原因 |

## 监控与审核注意点

- 建议监控指标：
  - 技能施放成功率
  - 技能失败原因分布
  - 移动端拖拽取消率
  - 连续输入被吞率
  - 弱网下技能失败率
- 容易被滥用的点：
  - 伪造目标点
  - 超距离施法
  - 篡改冷却
  - 高频请求轰炸

## 验收标准

- [ ] 玩家按下技能键后，客户端立刻出现技能反馈
- [ ] 服务端能稳定校验施法是否合法，并返回明确失败原因
- [ ] 同一种技能在 PC 和 Mobile 上都能稳定完成预瞄、确认和释放
- [ ] 技能的前摇、正式生效和冷却状态在视觉上可区分
- [ ] 输入缓冲可以工作，但不会无限堆积技能队列

## 未决问题

- MVP 阶段是否允许技能在前摇中手动取消
- 位移技能是否统一使用“确认时锁定方向”的规则
- 是否在早期就支持引导类技能，还是先做即时和单次释放技能
