# 对局流程

> **状态**：草稿
> **归属区域**：Server + Shared
> **最后更新**：2026-03-24

---

## 系统概述

对局流程系统负责管理一整场 MOBA 对局的生命周期，包括等待玩家、预备阶段、正式开局、对局中、结束结算和重置返回。它是所有战斗、刷兵、复活、UI 倒计时和胜负判定的上游状态机。

## 玩家体验目标

玩家应该始终知道比赛是否开始、什么时候能动、现在是否处于战斗阶段、为什么这局结束，以及多久能进入下一局。流程要稳定、明确、少歧义。

## 核心职责

- 管理整局比赛的状态流转
- 控制开局倒计时、刷兵起点、结束结算和回到房间
- 为其他系统提供统一的“当前对局状态”
- 在玩家不足、主基地被摧毁、时间结束等场景下决定是否结束比赛

## 输入

- 玩家加入与离开
- 玩家准备状态
- 主基地是否被摧毁
- 对局计时器
- 房间配置，如目标人数、倒计时、最大对局时长

## 输出

- 当前对局状态同步
- 开局倒计时事件
- 开放或关闭玩家控制
- 触发兵线系统开始工作
- 触发结算面板和结果广播

## 规则与状态变化

1. 默认状态为 `等待玩家`，当双方达到最小开局人数后进入 `预备阶段`。
2. `预备阶段` 持续固定倒计时，期间玩家可移动到出生区但不能对敌方造成有效战斗结果。
3. 倒计时结束进入 `正式对局`，同时开启刷兵、目标判定和成长系统。
4. 当任一方主基地被摧毁时，立即进入 `结算阶段`。
5. 如果达到最大时长且双方都未摧毁主基地，则进入 `加速结算` 或按当前目标优势直接判胜。
6. 结算阶段展示结果、KDA 和基础统计，短暂停留后回到 `等待玩家`。

## 服务端权威与复制策略

- 最终裁决方：Server
- 远程通信需求：
  - `MatchStateUpdated`
  - `MatchCountdownUpdated`
  - `MatchResultBroadcast`
- 反作弊校验点：
  - 客户端不能自行切换对局状态
  - 客户端不能伪造开局或结算事件
- 需要复制给客户端的状态：
  - 当前状态
  - 剩余倒计时
  - 剩余对局时间
  - 胜利方和结束原因

## 数据模型

| 字段 | 类型 | 持有方 | 说明 |
| --- | --- | --- | --- |
| MatchState | string | Server + Shared | 当前阶段，如 Waiting、Warmup、Running、Ending |
| CountdownRemaining | number | Server + Shared | 当前倒计时秒数 |
| MatchTimeRemaining | number | Server + Shared | 正式对局剩余时间 |
| TeamAScore | number | Server | 可选的辅助判胜数据 |
| TeamBScore | number | Server | 可选的辅助判胜数据 |
| EndReason | string | Server + Shared | 比赛结束原因 |

## 玩家交互流程

1. 玩家进入房间并等待人数满足开局条件
2. 屏幕显示倒计时和开局提示
3. 倒计时结束后进入正式对局
4. 玩家在对局中持续看到时间、局势和结果反馈
5. 主基地被摧毁或比赛超时后，显示结算并返回等待状态

## 边界情况

- 开局前有人退出导致人数不足
- 对局中有人掉线
- 同一帧发生主基地摧毁与时间结束
- 服务器重置或异常中断

## 可调参数

| 参数 | 默认值 | 安全范围 | 用途 |
| --- | --- | --- | --- |
| 最小开局人数 | 2v2 | 1v1 到 3v3 | 便于测试和正式对局切换 |
| 预备倒计时 | 10 秒 | 5 到 20 秒 | 让玩家有准备空间 |
| 最大对局时长 | 12 分钟 | 8 到 15 分钟 | 保持短局节奏 |
| 结算停留时间 | 8 秒 | 5 到 15 秒 | 让玩家看清结果后再重开 |

## 监控与审核注意点

- 建议监控指标：平均对局时长、对局中退出率、因人数不足取消的比例
- 容易被滥用的点：恶意卡准备、反复进出房间影响状态切换

## 验收标准

- [ ] 当人数满足条件时，比赛可以稳定从等待状态进入正式对局
- [ ] 主基地被摧毁后，比赛一定会结束并广播结果
- [ ] 客户端无法伪造开始、结束或胜利事件

## 未决问题

- MVP 阶段是否允许 1v1 和 2v2 作为调试模式存在
- 对局超时后的判胜规则是按塔数、主基地血量还是击杀数
