# 战斗结算

> **状态**：草稿
> **归属区域**：Server + Shared
> **最后更新**：2026-03-24

---

## 系统概述

战斗结算系统负责处理普攻、技能伤害、治疗、护盾、击杀归属和战斗反馈。它是 MOBA 核心体验的中心系统，决定“打中了没有”“这波谁赢了”“为什么会死”。

## 玩家体验目标

玩家应该觉得战斗结果公平、反馈清楚、死亡原因可理解。即便在 Roblox 的弱网环境下，也要尽量避免“明明躲开了却死了”或“明明打中了却没反应”的挫败感。

## 核心职责

- 校验普攻与技能请求是否合法
- 计算伤害、治疗、护盾和控制结果
- 维护战斗中的生命值变化
- 产生击杀、助攻和受击反馈事件
- 向成长系统输出金币与经验来源

## 输入

- 普攻请求
- 技能施放结果
- 目标状态，如是否存活、是否无敌、是否属于敌方
- 英雄属性数据
- 当前对局状态

## 输出

- 生命值变化
- 护盾变化
- 击杀与助攻事件
- 受击反馈和伤害飘字
- 死亡通知
- 奖励归属事件

## 规则与状态变化

1. 只有在 `正式对局` 状态下，战斗结算才会生效。
2. 服务端验证攻击发起者和目标都处于合法状态，如存活、敌对、在技能有效范围内。
3. 普攻遵循攻击间隔与攻击范围限制，不能超频触发。
4. 技能根据配置产生瞬时伤害、持续伤害、治疗、护盾或控制效果。
5. 当目标生命值小于等于 0 时，立即进入死亡流程，并向复活系统发出事件。
6. 击杀奖励优先归属最后一击来源，助攻归属在可配置时间窗内参与战斗的队友。

## 服务端权威与复制策略

- 最终裁决方：Server
- 远程通信需求：
  - `BasicAttackRequest`
  - `AbilityCastRequest`
  - `CombatEventBroadcast`
- 反作弊校验点：
  - 攻击速度、施法距离、目标合法性必须全部服务端验证
  - 客户端不能直接修改生命值、护盾值、伤害结果
- 需要复制给客户端的状态：
  - 当前生命值与护盾变化
  - 控制效果结果
  - 击杀播报
  - 伤害与治疗反馈

## 数据模型

| 字段 | 类型 | 持有方 | 说明 |
| --- | --- | --- | --- |
| Health | number | Server + Shared | 当前生命值 |
| MaxHealth | number | Shared | 最大生命值 |
| Shield | number | Server + Shared | 当前护盾值 |
| AttackPower | number | Shared | 基础攻击属性 |
| AbilityPower | number | Shared | 技能系数属性 |
| Defense | number | Shared | 减伤相关属性 |
| LastHitSource | string | Server | 最近致命伤害来源 |
| AssistLog | table | Server | 助攻追踪窗口 |

## 玩家交互流程

1. 玩家发起普攻或技能施放
2. 服务端校验行为是否合法
3. 若合法，则计算伤害、治疗或控制效果
4. 更新目标生命值、护盾和状态
5. 客户端收到反馈，显示受击、飘字、血条变化和击杀播报

## 边界情况

- 同时击杀
- 技能命中瞬间目标死亡
- 护盾与治疗同帧结算
- 目标离开可见范围但战斗结果仍需同步
- 玩家掉线时其召唤物或持续伤害效果仍在生效

## 可调参数

| 参数 | 默认值 | 安全范围 | 用途 |
| --- | --- | --- | --- |
| 助攻判定窗口 | 8 秒 | 4 到 12 秒 | 决定助攻归属 |
| 伤害反馈延迟容忍 | 0.1 秒 | 0 到 0.2 秒 | 控制表现与结算同步策略 |
| 普攻前摇比例 | 0.25 | 0.1 到 0.4 | 调整战斗节奏 |
| 基础死亡保护 | 0 秒 | 0 到 1 秒 | 避免极端连死体验 |

## 监控与审核注意点

- 建议监控指标：平均击杀时长、技能命中率、异常高伤玩家比例、同时击杀发生率
- 容易被滥用的点：伪造技能命中、超距离攻击、篡改冷却、异常高频请求

## 验收标准

- [ ] 所有伤害、治疗、护盾和击杀结果都由服务端最终决定
- [ ] 客户端无法通过伪造请求直接修改战斗结果
- [ ] 玩家能清楚看到受击、击杀和死亡反馈

## 未决问题

- MVP 阶段是否使用简化伤害公式，先不引入护甲穿透和复杂抗性
- 是否需要在早期就支持召唤物、持续区域伤害和多段技能
