# 游戏概念文档：裂隙先锋

*创建时间：2026-03-24*
*状态：草稿*

---

## 一句话介绍

> 《裂隙先锋》最终目标是一款 Roblox 上的短局动作 MOBA，但第一阶段不会先做峡谷推进，而是先做“英雄体验验证场”，让玩家在小型战斗空间中测试移动、技能、连招、拉扯和命中反馈。

## 核心定位

| 项目 | 内容 |
| --- | --- |
| 体验类型 | 动作 MOBA / 团队对抗 |
| 视角 | 第三人称 |
| 单局时长 | 8 到 12 分钟 |
| 玩家人数 | 3v3，后续可扩展为 5v5 |
| 平台 | PC + Mobile，优先保证跨平台可玩 |
| 变现方式 | 初期以免费体验为主，后续可扩展外观、英雄解锁、战令 |
| 目标规模 | 中小型持续迭代项目 |

## 玩家体验承诺

玩家应该稳定感受到三件事：技能按下就有响应、命中反馈清楚、英雄差异鲜明。正式 MOBA 结构可以后做，但英雄本身必须先“玩起来有劲”。

## 核心循环

### 秒级循环

玩家在走位、补兵、拉扯、释放技能、躲避技能与争夺站位之间反复切换，核心乐趣来自小规模博弈和即时反馈。

### 单局循环

原型阶段的单局循环是进入一个小型战斗场景，反复进行对拼、躲技能、追击、脱战、技能连招和死亡后快速重开。重点不是推线，而是持续验证英雄战斗节奏。

### 长线留存

玩家回来的原因主要包括：

- 英雄熟练度成长
- 不同英雄与阵容组合带来的新鲜感
- 赛季任务与短期目标
- 可追踪的战绩和进步反馈

## 核心系统

1. 英雄移动、朝向与目标选择
2. 技能释放、前摇、后摇、打断与冷却
3. 伤害结算、命中反馈、死亡与快速重开
4. 镜头、目标辅助与技能指示器
5. 英雄资源、状态异常与施法限制
6. 英雄原型测试场与训练 / 对拼流程

## 设计支柱

### 支柱 1：短局也要有 MOBA 感

虽然最终方向是 MOBA，但现阶段不追求完整地图结构，而是优先保留“走位、技能、压制、反打、击杀确认”这些最底层的英雄博弈体验。

### 支柱 2：操作反馈必须清楚

命中、受击、击杀、升级、抢目标这些关键事件必须一眼看懂，不能让玩家“输了也不知道怎么输的”。

### 支柱 3：先做可打磨的核心对抗，再扩外围系统

优先把英雄手感打透，再补地图目标、兵线、塔、防御结构、商城和复杂运营系统。

## 反向边界

- 不在第一阶段就做峡谷推进、兵线运营、防御塔博弈等完整 MOBA 地图结构
- 不在早期版本引入过深的局外养成和付费数值系统

## 参考体验

| 参考对象 | 借鉴什么 | 我们要做出的区别 |
| --- | --- | --- |
| League of Legends | 兵线、推塔、团战节奏和定位分工 | 对局更短，英雄技能更少更直接 |
| Pokémon Unite | 短局节奏、移动端可玩性、轻量成长 | 我们更强调塔防推进和传统 MOBA 目标 |
| Brawl Stars | 明快操作反馈和局内高频交互 | 我们保留更完整的兵线与目标争夺结构 |

## MVP 定义

MVP 目标不是验证整张 MOBA 地图，而是验证“这个英雄系统是否有手感、是否值得扩成 MOBA”。最小可玩版本包括：

- 一个小型战斗测试场
- 1 个原型英雄，后续扩展到 2 到 3 个
- 每个英雄拥有普攻和 3 个技能
- 可用的移动、朝向、目标辅助与技能预瞄
- 清晰的命中反馈、受击反馈、冷却与失败提示
- 死亡与快速重开
- 一个训练假人或简易对手目标
- 基础 HUD、技能按钮、血条和调试信息

MVP 阶段不要求：

- 峡谷地图
- 防御塔与主基地
- 兵线与小兵
- 装备商店
- 英雄 Ban / Pick
- 长线账号成长
- 正式匹配系统

## 当前原型阶段目标

第一阶段只回答下面几个问题：

- 技能按下去是不是立刻有响应
- 技能前摇和正式生效时刻是否清楚
- 走位、拉扯、技能释放和命中反馈是否顺滑
- 英雄之间未来是否有可扩展的定位差异
- Mobile 和 PC 在基础战斗操作上是否都能成立

## 运营与安全注意点

- 社交或聊天复杂度：基础
- 是否涉及高审核风险内容：低，主要风险来自玩家对喷和恶意引导
- 经济系统被刷取的风险：中，后续一旦接入金币、解锁和奖励就要服务端严格校验
- 存档与数据安全敏感度：中，后续如果加入英雄熟练度和赛季数据，风险会提高

## 技术备注

- 服务端权威模型：战斗结果、技能是否合法生效、死亡、状态异常和冷却由服务端最终裁决
- 是否需要持久化：原型阶段可以完全不做，先把英雄体验验证透
- 高风险集成项：技能判定与弱网表现、打断规则、位移技能、命中反馈、移动端操作适配
